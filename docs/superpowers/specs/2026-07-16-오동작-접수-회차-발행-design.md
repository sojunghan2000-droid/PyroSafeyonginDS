# '오동작 접수' 회차·Task 자동 발행 설계 (v1.10)

작성일: 2026-07-16
관련 요청: 작업 조치 관리 통합 목록에서 **직접 등록 오동작의 점검 ID·작업 ID가 "-"** 로 비는 문제 — 등록 시 새 점검/작업 ID를 발행해 추적 가능하게.

## 배경 / 문제

오동작(별지9)은 두 경로로 등록된다:
1. **점검 중 발생** — 점검 실행에서 결과 = "오동작" → `Malfunction.task_id` 채워짐 → 통합 목록에 점검 ID·작업 ID 정상 표시.
2. **직접 등록** — 안전점검 관리 우상단 [오동작 등록]([lib/inspection_dialog.py:1466](../../../lib/inspection_dialog.py)) → 소속 점검 회차/Task 없음 → `task_id=None` → 통합 목록에서 점검 ID·작업 ID가 **"-"** ([pages_app/deficiency_manager.py:87-88](../../../pages_app/deficiency_manager.py), [286-297](../../../pages_app/deficiency_manager.py)).

직접 등록 오동작 행이 **식별자 없이 "-"만** 있어 관리대장 추적이 곤란하다.

## 스코프 결정 (확정)

| 항목 | 결정 |
|---|---|
| 해결 방식 | 직접 등록 시 **실제 회차(Round)+Task 레코드를 새로 발행**하고 오동작에 연결 (문자열 ID만 부여 = 댕글링이라 금지) |
| 회차 유형 | 예약 유형 **"오동작 접수"** (정기 점검유형과 구분, 장비 inspection_types 카탈로그엔 미추가) |
| 회차 목록 노출 | **노출함** — "오동작 접수" 배지로 정기 점검과 구분 |
| 회차/Task 상태 | **Completed** (접수 이벤트는 완료 상태; 후속 조치는 오동작 자체 action_done으로 추적) |
| KPI | 오동작 접수 회차는 **점검 KPI에서 제외** |
| 별지5/6 | 회차 드롭다운에서 **제외** |
| 스키마 | **변경 없음** (기존 컬럼 사용) |

## 변경 1 — 오동작 등록 시 회차·Task 발행

**위치**: `lib/inspection_dialog.py` `malfunction_dialog` 제출 블록([1656-1677](../../../lib/inspection_dialog.py))

`add_malfunction` 호출 **전에** 회차 + Task 생성, Malfunction에 task_id 연결:

`InspectionRound / InspectionTask / add_round / add_task / next_round_id / next_task_id / Malfunction`은 [inspection_dialog.py:12-20](../../../lib/inspection_dialog.py)에 **이미 모듈 레벨 import** 되어 있어 접두사 없이 사용한다. 신규 상수만 `data.MAL_ROUND_TYPE`로 참조.

```python
    _loc = st.session_state.get("mal_dlg_loc_picked") or {}
    # 오동작 접수 회차 + Task 자동 발행 — 직접 등록분에 점검/작업 ID 부여
    _round_id = next_round_id()
    add_round(InspectionRound(
        round_id=_round_id, task_type=data.MAL_ROUND_TYPE,
        assignee=reporter, due_date=occurred, status="Completed",
        note=f"오동작 접수 · {category}",
    ))
    _task_id = next_task_id()
    _eq_label = (f"{_loc['floor']}/{_loc['zone']} · {category}"
                 if _loc else f"오동작 · {category}")
    add_task(InspectionTask(
        task_id=_task_id, equipment_label=_eq_label,
        task_type=data.MAL_ROUND_TYPE, assignee=reporter,
        due_date=occurred, status="Completed",
        floor=_loc.get("floor", ""), zone=_loc.get("zone", ""),
        round_id=_round_id,
    ))
    new_id = data.next_malfunction_id()
    add_malfunction(Malfunction(
        malfunction_id=new_id, category=category, occurred_on=occurred,
        detail=detail.strip(), action="", confirmer=reporter,
        action_done=False,
        floor=_loc.get("floor", ""), zone=_loc.get("zone", ""),
        spot_id=_loc.get("spot_id"),
        task_id=_task_id,          # ← 새 Task 연결
    ))
```

- 결과: 통합 목록 오동작 행의 `round_id`/`task_id`가 실값 → 점검 ID·작업 ID 표시.

## 엣지 케이스 / 비고

- **점검 중 발생 오동작**은 이미 task_id가 있어 이 변경의 영향 없음 (직접 등록분만 회차 발행).
- 회차 목록 상태 필터(전체/진행중/완료 등): 오동작 접수 회차는 Completed라 "완료" 또는 "전체"에 노출 — 의도된 동작.
- `round_progress`/진행바: Task 1/1 Completed → 100%. 정상.
- 오동작 접수 회차의 `task_type`이 카탈로그에 없어도 목록 렌더·상태계산엔 영향 없음(카탈로그는 장비 매핑용). 회차 목록 상단 점검유형 필터 콤보박스에 "오동작 접수"가 뜰 수 있음 — 뜨면 자연스러움(선택 시 해당 회차만 필터).

## 범위 밖 (YAGNI)

- [점검] 클릭 시 오동작 상세/작업 조치 관리로 점프(현재는 완료 Task 비활성 표시로 충분).
- 기존 직접 등록 오동작(task_id 없는 과거 데이터) 소급 회차 발행 — 신규 등록분부터 적용.
- 오동작 접수 회차 전용 KPI 카드 신설.
